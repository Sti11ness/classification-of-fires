## 0. Контекст проекта и исходные материалы

### 0.1. Назначение

Необходимо построить программный комплекс (ПК), реализующий функции экспертной системы (ЭС) как элемента СППР в режиме информационно-советующей системы (ИСС). ПК помогает ЛПР выбрать ресурсы (автотехника и др.) для ликвидации пожара. Итоговый продукт — **исполняемый файл .exe**, работающий в двух режимах: **Аналитик** (полнофункциональный) и **ЛПР** (ограниченный).

### 0.2. Источник данных

Входные данные — Excel-файлы с записями о пожарах, вручную внесёнными из «Карточек о пожарах». Состав полей относится к 8 разделам по приказу МЧС №625. В данных присутствуют искажения форматов/значений и пропуски.

### 0.3. Текущее состояние кода (что уже есть)

**Реализовано:**

* загрузка Excel, склейка листов, унификация колонок RU→EN;
* чистка/валидация (дубликаты, даты 2000–2020, парсинг времён, флаги качества, выбросы этажности, лог-ущерб);
* EDA (таблицы, графики, отчёт по пропускам);
* **Fire ES Desktop** — приложение на PySide6 с архитектурой MVVM + UseCase;
* Workspace — переносимая папка проекта;
* TaskRunner — фоновые задачи с прогрессом и отменой;
* Batch-прогноз — экспорт в Excel/CSV;
* Сборка .exe — pyinstaller.spec + build_exe.bat.

**Критическое несоответствие ТЗ:**

* текущий `rank_ref` вычисляется через severity (ущерб/потери), **а по ТЗ (2.5.1) ранг определяется по близости вектора ресурсов к нормативной таблице** (евклидово расстояние в нормированном пространстве).

## 2. Что нужно исправить в уже сделанном (обязательные изменения)

### 2.1. Исправить определение ранга (заменить `rank_ref`)

**Требование ТЗ 2.5.1:**
Ранг = ранг, соответствующий **минимальному евклидову расстоянию** между:

* **вектором ресурсов** (по типам автотехники), выделенных на пожар в записи;
* **нормативными векторами ресурсов** из нормативной таблицы (рис.3),
  после **нормирования компонент в [0;1]**.

**Что исправить:**

* Текущий `severity_score` и `rank_ref` оставить только как исследовательские признаки/отчёт (опционально), но **не использовать как “ранг по ТЗ”**.
* Ввести новое поле: `rank_norm` (или `rank_tz`) — ранг строго по нормативам.

### 2.2. Исправить “кластерный блок” под смысл ТЗ

В ТЗ “кластерный анализ” в режиме ранга — это фактически **разметка** по ближайшему нормативному прототипу (рангу), а не k-means по произвольным признакам.

* Кластеризацию по managed/observed можно оставить как дополнительный раздел исследования, но разметка ранга должна идти через нормативную таблицу ресурсов.

### 2.3. Привести UI к требованиям exe и 2 ролям

ipywidgets-UI в Jupyter не является exe и не удовлетворяет режимам Аналитик/ЛПР. Нужно вынести UI в отдельное приложение и упаковать в exe.

## 4.2. Режим 2.5.1: выходная переменная — РАНГ (по нормативам)

### 4.2.1. Нормативная таблица

* Загрузить нормативную таблицу (структура как рис.3) в `data/normative/normative_resources.csv`.
* В БД хранить её копию (`normatives`).

### 4.2.2. Разбор ресурсов из записи

Реализовать парсер поля(ей) ресурсов из исходных данных:

* вход: `equipment` (строка), `equipment_count` (общее), при наличии — дополнительные поля по стволам/средствам;
* выход: вектор `resource_vector` по категориям техники, согласованный с нормативной таблицей (например, `{"АЦ": 2, "АЛ": 1, ...}`).
* хранить распарсенный вектор:

  * как отдельные числовые колонки `res_AC`, `res_AL`, ... в `fires`,
  * и/или JSON-поле `resources_json` (допускается, но предпочтительнее колонки для обучения).

Парсер должен поддерживать:

* неоднородные форматы, опечатки, сокращения;
* “количество техники” без типов (в таком случае — пометка `resources_parse_quality='low'` и/или правила распределения не применять, оставить NaN).

### 4.2.3. Нормирование [0;1]

Определить и зафиксировать правило нормирования:

* базовое правило: для каждого типа техники `x_norm = x / max_normative_for_this_type` (clip to [0;1]).
* если в данных встречаются значения выше нормативных максимумов — клиппинг и флаг `flag_resource_over_norm`.

### 4.2.4. Расчёт ранга

Для каждого пожара:

* вычислить расстояние до каждого нормативного вектора ранга `r`:
  `dist_r = ||vec_norm - norm_table[r]||_2`
* `rank_tz = argmin_r dist_r`
* сохранить:

  * `rank_tz` (категория 1, 1-бис, 2, 3, 4, 5 — как в нормативе),
  * `distances` (опционально: все `dist_r` для UI и дебага).

### 4.2.5. Разметка выборки

В режиме 2.5.1 ПК должен уметь:

* разметить всю выборку `rank_tz`,
* показать отчёт по качеству парсинга и доле строк, где ранг определён.

## 4.5. Прогнозирование с набором вариантов и доверительными вероятностями (2.3.3)

### 4.5.1. Ранг (2.5.1)

* На вход: параметры пожара (входные признаки).
* На выход: top-K рангов с вероятностями (`predict_proba`), например K=3.
* UI должен показывать столбиковую диаграмму вероятностей по рангам (как на рис.3).

### 4.5.2. Ресурсы (2.5.2)

Требуется “набор вариантов” с доверительностью. Реализовать один из двух вариантов (выбрать первый как базовый):

1. **Бутстрап-ансамбль деревьев** (рекомендуется):

   * N переобучений дерева на бутстрап-выборках,
   * для каждого ресурса получить распределение прогнозов,
   * выдавать варианты P20/P50/P80 (или “минимальный/базовый/усиленный”) + числовой диапазон.
2. Альтернатива: Quantile Regression Forest не требуется (не дерево решений по ТЗ), поэтому не использовать как основной путь.

### 4.5.3. Связь с нормативным рангом

Для режима прогнозирования ресурсов дополнительно вычислять:

* “индуцированный ранг” по нормативной таблице для каждого сценария ресурсов,
  чтобы сравнивать режимы (3.3).

## 5. Цифровой двойник среды (3.1)

Реализовать генератор ИВД:

* генерирует выборку произвольного размера N,
* статистически близкую к исходной по распределениям и структуре пропусков,
* поддерживает инъекцию искажений (форматы, выбросы, пропуски, логические несоответствия).

Минимальный реализуемый подход:

* для категориальных: эмпирические распределения (с контролем редких категорий),
* для числовых: эмпирические распределения/квантили или параметрические (логнорм/гамма для ущерба),
* пропуски: отдельная модель “маски пропусков” (вероятности NaN по колонкам и зависимость от групп).
  Искажения:
* битые даты/время (строки нестандартного формата),
* экстремальные этажи/ущерб,
* несогласованность `fire_floor > building_floors`,
* некорректные коды/смешение типов.

Артефакты:

* `reports/tables/synth_quality_report.csv`
* `notebooks/05_digital_twin.ipynb` (сравнение распределений исходных/синтетических).

## 15. Политика качества модели (обновление от 2026-02-18)

Для повышения качества при больших пропусках зафиксированы следующие правила:

1. **Канонический источник для обучения:** лист `БД-1...2000--2020 (1+2)` (если в данных есть `source_sheet`, обучение фильтруется по нему).
2. **Основная метрика ранга:** `Macro-F1`.
3. **Основной online-набор признаков:** `online_tactical`:
   - `region_code`, `settlement_type_code`, `fire_protection_code`, `enterprise_type_code`,
   - `building_floors`, `fire_floor`, `fire_resistance_code`, `source_item_code`, `distance_to_station`,
   - `t_detect_min`, `t_report_min`, `t_arrival_min`, `t_first_hose_min`.
4. **Поля с 91%+ пропусков (`risk_category_code`, `fpo_class_code`)** исключаются из online value-фичей; при необходимости используются только индикаторы пропуска.
5. **Импутация для online-наборов по умолчанию:** специальный маркер `-1` (а не `0`).
6. **Запрещена утечка будущего** в online-модели: не использовать `fatalities`, `injuries`, `direct_damage`, `people_saved`, `people_evacuated`, `t_contained_min`, `t_extinguished_min`.
7. **Сравнение online-режимов обязательно:** `online_dispatch`, `online_early`, `online_tactical` (5-fold CV, с отчётом `accuracy`, `f1_macro`, `f1_weighted`).

Это ТЗ предназначено для передачи агенту (OpenHands/Cursor/иной) как единый документ требований и критериев готовности.
