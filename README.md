# Fire ES

Fire ES — desktop-приложение для классификации пожаров и поддержки принятия решений при оперативной оценке ситуации. Проект объединяет загрузку и очистку исторических карточек о пожарах, формирование локальной рабочей базы, обучение модели `rank_tz` и использование этой модели в интерфейсе для аналитика и ЛПР.

Основной пользовательский сценарий выглядит так:

1. аналитик импортирует Excel-файлы с историческими записями о пожарах;
2. система очищает и валидирует данные, сохраняет их в локальный Workspace;
3. аналитик обучает или переобучает модель ранга пожара;
4. ЛПР вводит оперативные параметры пожара и получает прогноз ранга с вероятностями;
5. итоговое решение фиксируется в локальной базе и попадает в историю решений.

## Что делает приложение

- работает как офлайн desktop-приложение для Windows;
- поддерживает два режима работы: `Аналитик` и `ЛПР`;
- импортирует Excel-данные о пожарах, очищает и валидирует их;
- хранит данные, модели, отчеты и логи в переносимом Workspace;
- обучает модели для прогноза `rank_tz`;
- показывает Top-K вариантов ранга с оценкой уверенности;
- поддерживает пакетный прогноз по Excel-файлу;
- сохраняет решения ЛПР и историю прогнозов.

## Для каких данных предназначен проект

Проект работает с табличными данными о пожарах, поступающими из Excel-файлов. В коде и документации используется структура, близкая к карточкам о пожарах и разделам 1–7 прикладной схемы проекта:

- общие сведения о пожаре;
- параметры объекта и места пожара;
- последствия;
- сведения о спасенных и эвакуированных;
- времена реагирования;
- сведения о привлеченных ресурсах;
- дополнительные атрибуты для анализа и прогноза.

После импорта данные приводятся к унифицированной схеме, очищаются и сохраняются в SQLite внутри рабочей папки проекта.

## Как в проекте понимается ранг пожара

Основная целевая переменная проекта — `rank_tz`.

В текущей реализации canonical production-контур режима `2.5.1` опирается на `rank_tz_vector`: ранг вычисляется по минимальному евклидову расстоянию между нормированным вектором ресурсов пожара и нормативным вектором. `rank_tz_count_proxy` сохранен только как auxiliary/legacy label и не может быть активирован как canonical ЛПР-модель. Подробная инженерная логика, ограничения и соответствующие модули описаны в [техническом README](docs/TECHNICAL_README.md) и [ML methodology](docs/ML_METHODOLOGY.md).

## ML-часть на уровне продукта

Система поддерживает обучение моделей для прогноза `rank_tz` на исторических данных. В репозитории есть как исследовательский ML-слой, так и desktop pipeline для прикладного использования.

На уровне пользовательского продукта важно следующее:

- доступны модели на основе дерева решений и случайного леса;
- рабочий desktop pipeline ориентирован на `rank_tz`;
- для прогноза используются признаки, доступные в оперативном сценарии;
- результат выдается не одной меткой, а набором лучших вариантов с вероятностями;
- для batch-сценария результаты могут быть выгружены в Excel или CSV.

## Режимы работы

### Аналитик

Режим предназначен для обслуживания системы и работы с данными:

- создание и открытие Workspace;
- импорт и очистка данных;
- обучение модели;
- пакетный прогноз;
- просмотр реестра моделей;
- просмотр журнала операций;
- тестовый запуск экрана прогноза.

### ЛПР

Режим предназначен для оперативного использования:

- ввод входных параметров пожара;
- получение прогноза ранга с вероятностями;
- просмотр нормативной таблицы;
- сохранение решения и комментария;
- просмотр журнала и истории решений.

## Структура решения верхнего уровня

Проект состоит из двух основных прикладных контуров:

- `src/fire_es` — domain/ML-логика: очистка, ранжирование, обучение, прогнозирование, схема БД;
- `src/fire_es_desktop` — desktop-приложение на PySide6: Workspace, Use Cases, ViewModels, UI, реестр моделей, логи.

Дополнительно в репозитории есть:

- исследовательские скрипты и артефакты в корне репозитория;
- тесты `pytest`;
- документы в `docs/`;
- сценарии сборки и запуска в `scripts/`.

## Быстрый старт

### Запуск из исходного кода

```bash
pip install -e .
pip install PySide6 plotly

python -m fire_es_desktop.main --role analyst
python -m fire_es_desktop.main --role lpr
```

### Быстрый запуск через bat-скрипт

```bash
scripts\run_desktop_dev.bat analyst
scripts\run_desktop_dev.bat lpr
```

## Что сохраняется в Workspace

Рабочая папка проекта содержит:

- `fire_es.sqlite` — локальную базу данных;
- `config.json` — конфигурацию Workspace;
- `reports/models/` — модели и связанные метаданные;
- `reports/figs/` — графики;
- `reports/tables/` — табличные выгрузки;
- `logs/` — прикладные и task-логи.

Такой формат позволяет переносить проект как папку целиком.

## Ограничения и важные замечания

- основной desktop pipeline сейчас ориентирован на задачу `rank_tz`;
- рабочий production-контракт признаков для ЛПР отделен от offline-экспериментов;
- ранжирование по нормативной логике в текущей рабочей версии опирается на упрощенное представление ресурсов по `equipment_count`;
- production-метрики считаются только при leakage-safe split c паспортом `semantic_target + label_source_policy + availability_stage + split_protocol + event_overlap_rate`;
- в репозитории одновременно живут desktop workflow и исследовательские артефакты, поэтому технические детали лучше смотреть в инженерной документации.
